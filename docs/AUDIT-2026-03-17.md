## CRITICAL Findings

### C1. `/api/chat/confirm` has no authentication — unauthenticated calendar write path

**File:** `backend/app/agents/router.py:195`
**Found by:** Backend Core review + Agent Pipeline review (independently confirmed)

**Problem:**
```python
user_id = "dev-user"  # stub until auth wired in #9/#10/#11
```
The referenced issues (#9, #10, #11) merged months ago. The `chat` endpoint correctly uses `Depends(get_current_user)`, but `confirm` does not. Any unauthenticated caller can:
1. Supply a thread_id starting with `"user-dev-user:session-"`
2. Pass the ownership check on line 197
3. Execute calendar create/update/delete operations on behalf of any user

**Impact:** Structural auth bypass on the only endpoint that triggers calendar mutations.

**Fix:**
- Add `Depends(get_current_user)` to the confirm endpoint
- Derive `user_id` from the authenticated user object
- Update existing tests in `TestConfirmEndpoint` to inject auth

**Complexity:** Small — mirrors the pattern already used on the `chat` endpoint.

**Decision:** Group with C2 + C3 into a single bug issue — all three are backend security/reliability bugs that can be fixed in one pass.
**Issue:** #100

### C2. Rate limiter is IP-based, not user-based — spec violation

**File:** `backend/app/core/middleware.py:13`
**Found by:** Backend Core review

**Problem:**
```python
limiter = Limiter(key_func=get_remote_address, default_limits=["60/minute"])
```
SPEC Security Constraint #9 requires "20 requests/minute per user on chat endpoint."
Two problems:
1. `get_remote_address` keys by IP, not authenticated user — users behind NAT share a bucket; attackers rotate IPs to bypass
2. No `@limiter.limit("20/minute")` decorator on the chat route — it inherits the 60/min global default

**Impact:** Rate limiting is ineffective against targeted abuse and diverges from spec.

**Fix:**
- Create a custom key function that extracts `user_id` from the authenticated session
- Add `@limiter.limit("20/minute")` to the chat endpoint specifically
- Keep the global 60/min default for unauthenticated endpoints (health, etc.)

**Complexity:** Small-medium — need to wire auth into the rate limiter key function.

**Decision:** Group with C1 + C3 — same issue.
**Issue:** (see grouped issue)

### C3. `_refresh_locks` unbounded memory leak

**File:** `backend/app/agents/tools/calendar_tools.py:46-53`
**Found by:** Agent Pipeline review

**Problem:**
```python
_refresh_locks: dict[str, asyncio.Lock] = {}
```
`asyncio.Lock` objects are added per `user_id` and never evicted. In a long-running process serving many distinct users, this grows without bound.

**Impact:** Slow memory leak proportional to distinct user count. Container eventually OOMs under sustained multi-user load.

**Fix options:**
1. `cachetools.LRUCache(maxsize=1000)` — bounded, evicts least-recently-used
2. `weakref`-based approach — evict when no references remain
3. Evict on lock release — delete key after `async with lock` block exits

**Complexity:** Small — replace dict with bounded cache.

**Decision:** Group with C1 + C2 — same issue.
**Issue:** (see grouped issue)

### C4. Confirmation UI never rendered — human-in-the-loop is broken end-to-end

**File:** `frontend/src/components/chat/ChatMessage.tsx`
**Found by:** Frontend UI review

**Problem:**
`useChat.ts` correctly manages `pendingConfirmation` state and exposes `confirmAction(approved: bool)`. But `ChatMessage.tsx` only renders text content bubbles and typing indicators — no Confirm/Cancel button pair exists anywhere in the UI.

Combined with C1 (confirm endpoint has no auth), human-in-the-loop is **non-functional at every layer**:
- Frontend: no UI to approve/reject
- Backend: no auth on the confirm endpoint

**Impact:** Calendar write operations (create/update/delete) silently hang waiting for approval that can never arrive through the UI.

**Fix:**
- Add a `ConfirmationCard` component rendered inside `ChatMessage` when the message contains a confirmation payload
- Show action details (event name, time, type of operation) + Approve/Reject buttons
- Wire buttons to `confirmAction` from `useChat`

**Complexity:** Medium — UI component + integration with existing hook state.

**Decision:** Standalone issue — this is a frontend feature with its own acceptance criteria and UI design work, separate from the backend bugs in C1-C3.
**Issue:** #101

## WARNING Findings

### W1. `GET /api/auth/session` endpoint missing

**File:** `backend/app/auth/router.py`
**Found by:** Auth review

**Problem:** SPEC defines `GET /api/auth/session` to return current session info. No such endpoint exists in the backend auth router. Session management is handled entirely by Auth.js on the frontend side.

**Analysis:**
The backend doesn't own sessions — Auth.js does. The backend is zero-trust: it verifies the Google ID token on every request via `get_current_user` and holds encrypted OAuth tokens in Redis for Google API calls. The frontend already knows auth state (session cookie, user profile, granted scopes) from Auth.js. The only backend-specific info (token validity in Redis) surfaces naturally as 401/scope errors when the agent is invoked.

**Decision:** Spec artifact — remove `GET /api/auth/session` from the SPEC API contracts. No implementation needed. The zero-trust per-request verification model makes a dedicated session endpoint redundant.
**Action:** Update SPEC.md to remove this endpoint from the Auth Endpoints table.

---

### W2. `POST /api/auth/consent` endpoint missing

**File:** `backend/app/auth/router.py`
**Found by:** Auth review

**Problem:** SPEC defines `POST /api/auth/consent` to trigger incremental consent for new scopes. The frontend handles consent redirects via Auth.js, but there's no backend endpoint for it.

**Analysis:**
Incremental consent is already fully implemented without this endpoint. The flow:
1. Settings page `ScopeManager` → `signIn("google", ..., { scope: "base + new" })`
2. Auth.js with `include_granted_scopes: "true"` → Google merges scopes
3. JWT callback captures merged token + scope list
4. `syncTokenToBackend()` POSTs to the **existing** `POST /api/auth/callback`
5. Backend stores updated scopes + tokens in Redis
6. Settings page re-renders from `GET /api/users/me` showing new scopes

The existing `/api/auth/callback` handles both initial sign-in and incremental consent — same OAuth flow, same endpoint. A dedicated `/api/auth/consent` endpoint would be redundant.

**Decision:** Spec artifact — remove `POST /api/auth/consent` from the SPEC API contracts. Consent is handled by the standard OAuth callback flow. No implementation needed.
**Action:** Update SPEC.md to remove this endpoint and add a note that incremental consent reuses `/api/auth/callback`.

### W3. Google auth transport session never closed on shutdown

**File:** `backend/app/auth/dependencies.py:34`
**Found by:** Auth review

**Problem:** `CacheControl(requests.Session())` wraps a sync HTTP session, lazily initialized on first auth request. Used for Google cert fetching in `verify_oauth2_token`. The `run_in_executor` call on line 60 correctly offloads the blocking I/O, so async safety is handled. The real issue: the session is never closed on shutdown — TCP connection pool and cached certs leak.

**Impact:** Resource leak on process shutdown. Minor in practice (OS reclaims on exit) but violates the project's own rule: "Every resource opened in startup MUST be closed in shutdown."

**Fix (small, ~10 lines):**
1. Add `close_google_transport()` to `dependencies.py` that calls `_google_transport.session.close()` and resets the singleton
2. Call it from the lifespan shutdown in `main.py` alongside `close_redis()`, `close_search_client()`, etc.
3. Follows the exact same cleanup pattern already used for Redis and search clients

**Complexity:** Small — isolated change, no behavioral impact, follows existing patterns.

**Decision:** Group with W6 + W7 + W8 into a single backend hardening issue — all are small, independent fixes that can be done in one pass.
**Issue:** #102

### W9. `MemorySaver` in all environments — no production checkpointer

**File:** `backend/app/agents/calendar_agent.py:71`
**Found by:** Agent Pipeline review

**Problem:** SPEC Decisions Log says "InMemorySaver for dev, PostgresSaver for prod." Only `MemorySaver` is ever instantiated. On container restart (rolling deploy, scale-in), all conversation state is lost. The human-in-the-loop `Command(resume=True)` will raise because the thread checkpoint no longer exists.

**Impact:** Multi-turn conversations and pending confirmations break silently after any restart.

**Analysis — Redis checkpointer is the right path:**
`langgraph-checkpoint-redis` v0.4.0 exists with `AsyncRedisSaver`. The swap is minimal:
1. `uv add langgraph-checkpoint-redis` — one new dependency
2. Replace `MemorySaver()` with `AsyncRedisSaver.from_conn_string(settings.redis_url)` in `calendar_agent.py` (~3 lines)
3. Call `.asetup()` in app lifespan to create Redis indices
4. Both checkpointer and existing token storage coexist on the same Redis instance — no new infrastructure needed

All thread ID generation, config passing, and agent pipeline wiring is already in place. It's a storage backend swap, not an architecture change.

**Decision:** Defer to Phase 2. Accept `MemorySaver` for MVP — conversations don't survive restarts, which is a known limitation. For Phase 2, use `langgraph-checkpoint-redis` (not PostgresSaver) since Redis is already deployed and this avoids adding a new database dependency. Update SPEC Decisions Log to reflect this change: "MemorySaver for MVP; Redis checkpointer (`langgraph-checkpoint-redis`) for Phase 2 — reuses existing Redis, no new infrastructure."
**Issue:** Phase 2 backlog item, not MVP.

### W13. `SyncMetadata` missing `window_start` and `window_end` fields

**File:** `backend/app/context_ingestion/sync.py:28`
**Found by:** Search & Ingestion review

**Problem:** SPEC defines 4 fields; implementation only stores `sync_token` and `last_ingested_at`. Missing fields were intended for Phase 2 cleanup queries.

**Analysis:**
The Phase 2 cleanup job described in the SPEC deletes documents where `timestamp < (now - 7 months)` — that query uses the `timestamp` field on search documents, not `window_start`/`window_end` on sync metadata. The ingestion window is derived at runtime from `datetime.now()` + fixed constants (`_PAST_WINDOW_DAYS=183`, `_FUTURE_WINDOW_DAYS=91`) and recalculated on every full ingest. No code reads stored boundaries to decide what to fetch. The only scenario where stored boundaries would matter is detecting "this user's index only covers 3 months back, let's extend it" — a feature that isn't planned.

**Decision:** Spec artifact — remove `window_start` and `window_end` from the `SyncMetadata` model in the SPEC. The fields were designed speculatively for a cleanup job that doesn't need them.
**Action:** Update SPEC.md to reduce `SyncMetadata` to `sync_token` + `last_ingested_at` only.

### W16. SSE streaming and confirmation bypass `api.ts`

**File:** `frontend/src/hooks/useChat.ts`, `frontend/src/actions/chat.ts`
**Found by:** Frontend UI review

**Problem:** SPEC says "ALL backend calls through `frontend/src/lib/api.ts`." The streaming path uses `@/lib/chat-stream` and confirmations use Server Actions — both bypass `api.ts`.

**Analysis:**
Both "bypasses" are architecturally valid:
1. **SSE streaming** — `api.ts` uses standard `fetch` with `await response.json()`, designed for request/response. SSE streams are long-lived and can't pipe through that pattern. `chat-stream` is the correct abstraction.
2. **Server Actions** — `@/actions/chat` handles confirmations server-side, adding auth tokens and calling the backend. This is the correct Next.js pattern for authenticated mutations from client components.

The SPEC rule was written before streaming and Server Actions patterns were implemented.

**Decision:** Not a spec artifact — this is a real bug. SSE stream goes Browser → Backend directly via `NEXT_PUBLIC_API_URL` (internal FQDN), which the browser can't resolve in production. Chat is broken in deployed environments. Issue #99 fixes this by routing SSE through a Next.js server-side proxy. Once #99 is resolved, the SPEC rule "all backend calls through api.ts" becomes correct — all traffic flows Browser → Next.js (public) → Backend (internal).
**Issue:** #99 (already created, critical blocker for all other bug fixes)
**SPEC updated:** Frontend API rule now documents the full server-side proxy architecture.
